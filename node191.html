การติดต่อสื่อสารกับหน่วยประมวลผล

กระบวนการตรวจสอบบิตแสดงสถานะแบบเป็นครั้งคราวเพื่อดูว่าถึงเวลาสำหรับการทำงานของ I/O ตัวถัดไปหรือยัง ที่เรียกว่า polling ซึ่งเป็นวิธีพื้นฐานที่สุดสำหรับอุปกรณ์ I/O ในการติดต่อกับหน่วยประมวลผล อุปกรณ์ I/O ทั่วไปจะใส่ข้อมูลลงใน Status register และหน่วยประมวลจะมานำข้อมูลไปใช้ หน่วยประมวลผลจะทำงานครอบคลุมทั้งในส่วนของการควบคุมและกระบวนงานทั้งหมด

ข้อเสียของ polling คือทำให้เกิด processor time ที่เสียเปล่าจำนวนมากเพราะว่าหน่วยประมวลมีความเร็วมากกว่าอุปกรณ์ I/O หน่วยประมวลผลอาจจะอ่าน Status register หลายครั้งเพียงเพราะอุปกรณ์ยังทำงานไม่เสร็จสมบูรณ์

ปัญหาที่เกิดขึ้นได้นำไปสู่แนวความคิดของการใช้ interrupt เพื่อแจ้งหน่วยประมวลผลเมื่ออุปกรณ์ I/O มีการร้องขอไปยังหน่วยประมวลผล Interrupt-driven I/O ซึ่งถูกใช้ในระบบเกือบทั้งหมดสำหรับบางอุปกรณ์ จะเป็นตัวเรียกใช้ I/O interrupt เพื่อบอกหน่วยประมวลผลว่าอุปกรณ์ I/O ต้องการเข้าใช้งาน หรือเพื่อบอกว่าทำงานสมบูรณ์แล้ว จะเป็นสาเหตุให้หน่วยประมวลผลเกิดการ interrupt ขึ้น

I/O interrupt จะเหมือนกับ exceptions ที่เราพบในบทที่ 5, 6 และ 7 ที่มีส่วนสำคัญ 2 ส่วนคือ

  1. I/O interrupt เป็น asynchronous ขึ้นอยู่ instruction execution โดย control unit ใช้เพียงแค่ตรวจสอบสำหรับ I/O interrupt ที่ค้างอยู่ ขณะที่เริ่ม instruction ใหม่
  2. ในความเป็นจริงเมื่อเกิด interrupt ขึ้น เราอยากที่จะส่งข้อมูล เช่น ID อุปกรณ์ที่สร้าง interrupt นอกจากนี้ interrupt ได้แสดงถึงลำดับความสำคัญของอุปกรณ์ที่อาจจะแตกต่างกัน

ในการส่งข้อมูลไปยังหน่วยประมวลผล เช่น ID อุปกรณ์ที่ทำให้เกิด interrupt ระบบสามารถใช้ทั้ง vectored interrupt หรือ exception Cause register เมื่อหน่วยประมวลผลพบการ interrupt อุปกรณ์สามารถส่ง vector address หรือ status field เข้าไปใน Cause register ผลลัพธ์ที่ได้ เมื่อระบบปฏิบัติการทำการควบคุมก็จะรู้ว่าอุปกรณ์ตัวไหนที่เกิด interrupt และสามารถติดต่อกับอุปกรณ์ได้ทันที กลไกการ interrupt ได้ยกเลิกความจำเป็นสำหรับหน่วยประมวลผลในการ poll อุปกรณ์ และอนุญาตให้หน่วยประมวลผลไป focus บน โปรแกรมที่กำลังทำงานอยู่แทน
